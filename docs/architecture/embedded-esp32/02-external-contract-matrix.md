# External Contract Matrix And Decision Closure

This file captures every external assumption that can sink the embedded program. Every assumption row is intentionally marked `HYPOTHESIS` until its owning phase produces live evidence.

Known platform and toolchain constraints that are already verified outside the repo live in [09-technical-dependency-analysis.md](./09-technical-dependency-analysis.md). This file is only for assumptions that still require phase-owned proof.

## Test environments

| Environment ID | State | Definition | Use |
| --- | --- | --- | --- |
| ENV-ESP32-HW | `HYPOTHESIS` | Real ESP32-S3 hardware with PSRAM, USB serial access, reachable Wi-Fi, provider credentials, and a repeatable flash/monitor workflow. An 8MB-class PSRAM board is the planning baseline; smaller PSRAM variants are only admitted if Phase 0 metrics explicitly clear them. | Only this environment can close critical assumptions. |
| ENV-ESP32-SWARM-HW | `HYPOTHESIS` | A small real-node ESP32-S3 rig on the same Wi-Fi network with stable node identities, known serial ports, and a repeatable physical placement or inventory file. Use the same 8MB-class PSRAM planning baseline unless later evidence relaxes it. | Required to close swarm and relative-position assumptions. |
| ENV-ESP32-SIM | `HYPOTHESIS` | An ESP32-compatible simulator or emulator, if available, with the same harness commands and marker protocol. | Supplemental evidence only. Useful for harness debugging and non-closing CI. |
| ENV-HOST-SIM | `HYPOTHESIS` | A desktop host simulator that exercises the same embedded surface API and smoke marker protocol without claiming target-hardware closure. | Fast TDD support and negative controls. |

## Planning-time decision closure

These rows close the architecture choices that the implementation is not allowed to renegotiate during execution.

| Decision ID | Decision | Basis | Consequence |
| --- | --- | --- | --- |
| DEC-001 | The required-scope baseline is a native ESP surface, not an on-device WASM authority path. | `BASE-006`, `BASE-008`, `BASE-009`, `ASSUMP-008` | Phase 2 builds a native Rust ESP surface and backend. Any shared embedded crate must be justified by real surface glue extracted in Phase 1 rather than speculative abstraction. A WAMR experiment is not part of the required scope. |
| DEC-002 | Provider protocol logic stays shared, but "shared" is defined at the level of request construction, incremental response parsing, retry semantics, and normalized event mapping. The transport, TLS backend, HTTP client, and sync/async bridge may vary by platform. | `BASE-002`, `BASE-008`, `BASE-014`, `BASE-015`, `SEAM-004` | Phase 1 must introduce a transport seam inside `meerkat-client` without pretending the ESP path is a byte-for-byte clone of the current `reqwest` implementation. |
| DEC-003 | Embedded must use the runtime-backed semantic path by default. | `BASE-003`, `BASE-004`, `BASE-005` | Phase 2 may start with an ephemeral runtime driver if needed, but not with a browser-style substrate-authority model. |
| DEC-004 | Embedded capability limits are explicit profile limits and backend limits, not reduced semantics. | `BASE-010`, `docs/reference/session-contracts.mdx#L11-L29` | Unsupported capabilities must fail deterministically through normal contracts instead of disappearing into a shadow implementation. |
| DEC-005 | Host-tool callbacks and fire-and-forget acknowledgement are part of the required embedded contract. | `BASE-007`, `SEAM-005` | Phase 1 extracts shared host-tool glue; Phase 2 binds it to ESP host callbacks; Phase 3 proves it in `examples/036` and `examples/037`. |
| DEC-006 | Public examples are user-facing recommended patterns first; internal contract probes live under `scripts/live_smoke/*` instead of taking over the examples tree. | `CONVENTION-001`, `CONVENTION-003`, user-direction for this dossier | Phase 0 builds an internal probe harness; Phase 3 examples optimize for reusable patterns while still providing smoke modes. |
| DEC-007 | Phase 0 may use the fastest sacrificial board-native probe stack that closes board and network contracts, but those results do not close Rust-stack-specific rows by themselves. | Adjacent field learning from [08-field-learnings-from-m5dial.md](./08-field-learnings-from-m5dial.md), [09-technical-dependency-analysis.md](./09-technical-dependency-analysis.md), plus `CHOKE-001` | Autonomous agents may use PlatformIO/Arduino, ESP-IDF C, or another fast vendor-native path for probes, but they must rerun toolchain, executor, and Rust-stack envelope checks on the planned Rust path before Phase 1 starts. |
| DEC-008 | Phase 0 must close every Phase-0-owned High-criticality assumption before Phase 1 starts. | `ASSUMP-001`, `ASSUMP-002`, `ASSUMP-003`, `ASSUMP-004`, `ASSUMP-008`, `ASSUMP-012` | If any Phase-0-owned High row is still open, the plan must be revised before architecture work continues. |
| DEC-009 | Example 037 has a predeclared fallback: if metric RSSI or timing positioning is rejected on real hardware, the required scope becomes a confidence-scored topology or adjacency self-organization artifact produced from the same real run inputs and telemetry. | `ASSUMP-011`, `REQ-011`, user-direction for this dossier | The autonomous implementation run may not reopen product scope if metric positioning fails. It must execute the fallback contract already defined in the dossier. |
| DEC-010 | Example 037 has no substitute fallback if basic peer discovery or bounded-latency messaging is rejected on the real Phase 3 stack. | `ASSUMP-010`, `REQ-011`, `CHOKE-005` | If `ASSUMP-010` is `REJECTED`, Example 037 cannot ship in required scope. The autonomous run must stop with an explicit program-stop record or replan outside this dossier instead of inventing a replacement swarm pattern. |

## Phase 0 completion rule for this matrix

Phase 0 is complete only when:

- every Phase-0-owned `High` row is `LIVE_VALIDATED` or the program is explicitly stopped and replanned;
- every Phase-0-owned `Medium` row is at least `PROVISIONAL`, with a concrete path to `LIVE_VALIDATED` inside required scope;
- this file is updated with the artifact paths and verdicts produced by the real probes;
- swarm-owned rows remain open until Phase 3 rather than being forced into the baseline board-feasibility phase.

# Realtime

The previous public realtime surface is retired while the new live adapter
architecture is rebuilt.

Removed surfaces:

- JSON-RPC `realtime/open_info`, `realtime/status`, `realtime/capabilities`
- JSON-RPC `session/realtime_attachment_status`
- REST `/realtime/open_info`, `/realtime/status`, `/realtime/capabilities`
- REST `/sessions/{id}/realtime-attachment-status`
- MCP `meerkat_realtime_open_info`
- CLI `rkat realtime ...`
- Python and TypeScript SDK realtime helper methods

The retired model treated realtime as a model-capability-driven session
attachment and exposed a websocket URL as the product bootstrap. That model was
not a truthful owner boundary: provider session lifecycle, websocket transport
state, runtime turn admission, tool dispatch, and mob identity were all coupled.

The replacement design is an internal live adapter seam:

- provider adapters own provider mechanics
- Meerkat owns admitted user input, cancellation, steering, tools, session
  history, mob identity, auth, and realm checks
- adapter observations are typed facts that a runtime-owned host gates into
  canonical Meerkat APIs
- transport bootstrap is tagged so websocket, WebRTC, and future provider
  transports do not define Meerkat semantics

Until a new `live/*` or MobKit-specific API is published, applications should
not rely on the retired realtime API.

# Phase B - External Contract Matrix And Decision Closure

This file captures every external assumption that can sink the embedded program. Every assumption row is intentionally marked `HYPOTHESIS` until a Phase 0 live contract test updates its status.

## Test environments

| Environment ID | State | Definition | Use |
| --- | --- | --- | --- |
| ENV-ESP32-HW | `HYPOTHESIS` | Real ESP32-S3 hardware with PSRAM, USB serial access, reachable Wi-Fi, provider credentials, and a repeatable flash/monitor workflow. | Only this environment can close critical assumptions. |
| ENV-ESP32-SWARM-HW | `HYPOTHESIS` | A small real-node ESP32-S3 rig on the same Wi-Fi network with stable node identities, known serial ports, and a repeatable physical placement or inventory file. | Required to close swarm and relative-position assumptions. |
| ENV-ESP32-SIM | `HYPOTHESIS` | An ESP32-compatible simulator or emulator, if available, with the same harness commands and marker protocol. | Supplemental evidence only. Useful for harness debugging and non-closing CI. |
| ENV-HOST-SIM | `HYPOTHESIS` | A desktop host simulator that exercises the same embedded surface API and smoke marker protocol without claiming target-hardware closure. | Fast TDD support and negative controls. |

## Planning-time decision closure

These rows close the architecture choices that the implementation is not allowed to renegotiate during execution.

| Decision ID | Decision | Basis | Consequence |
| --- | --- | --- | --- |
| DEC-001 | The required-scope baseline is a native ESP surface, not an on-device WASM authority path. | `BASE-006`, `BASE-008`, `BASE-009`, `ASSUMP-008` | Phase 2 builds `meerkat-embedded-runtime` and `meerkat-esp-runtime` as native Rust surface/backend crates. A WAMR experiment is not part of the required scope. |
| DEC-002 | Provider protocol logic stays shared; Phase 1 must introduce a transport seam inside `meerkat-client` instead of forking provider implementations. | `BASE-002`, `BASE-008`, `SEAM-004` | All provider-related embedded work is blocked on the transport seam. |
| DEC-003 | Embedded must use the runtime-backed semantic path by default. | `BASE-003`, `BASE-004`, `BASE-005` | Phase 2 may start with an ephemeral runtime driver if needed, but not with a browser-style substrate-authority model. |
| DEC-004 | Embedded capability limits are explicit profile limits and backend limits, not reduced semantics. | `BASE-010`, `docs/reference/session-contracts.mdx#L11-L29` | Unsupported capabilities must fail deterministically through normal contracts instead of disappearing into a shadow implementation. |
| DEC-005 | Host-tool callbacks and fire-and-forget acknowledgement are part of the required embedded contract. | `BASE-007`, `SEAM-005` | Phase 1 extracts shared host-tool glue; Phase 2 binds it to ESP host callbacks; Phase 3 proves it in `examples/036` and `examples/037`. |
| DEC-006 | Public examples are user-facing recommended patterns first; internal contract probes live under `scripts/live_smoke/*` instead of taking over the examples tree. | `CONVENTION-001`, `CONVENTION-003`, user-direction for this dossier | Phase 0 builds an internal probe harness; Phase 3 examples optimize for reusable patterns while still providing smoke modes. |
| DEC-007 | Phase 0 may use the fastest sacrificial board-native probe stack that closes external contracts; this does not bind the Phase 2 production implementation stack. | Adjacent field learning from [08-field-learnings-from-m5dial.md](./08-field-learnings-from-m5dial.md) plus `CHOKE-001` | Autonomous agents may use PlatformIO/Arduino, ESP-IDF C, or another fast vendor-native path for probes, then still implement the production runtime in the planned stack. |
| DEC-008 | Phase 0 must close every High-criticality assumption before Phase 1 starts. | `ASSUMP-001` through `ASSUMP-011` | If any High-criticality row is still open, the plan must be revised before architecture work continues. |

## Phase 0 completion rule for this matrix

Phase 0 is complete only when:

- every `High` row is `LIVE_VALIDATED` or the program is explicitly stopped and replanned;
- every `Medium` row is at least `PROVISIONAL`, with a concrete path to `LIVE_VALIDATED` inside required scope;
- this file is updated with the artifact paths and verdicts produced by the real probes.
